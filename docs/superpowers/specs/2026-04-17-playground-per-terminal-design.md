# Per-terminal playground ticket toggle

**Date:** 2026-04-17
**Scope:** `managers` (backend + admin) and `tablo_full/new_webserver` (iiko plugin — no code changes)

## Problem

The iiko plugin (`new_webserver`) calls `POST /api/playground_tickets/generate` on every paid order whose sum meets `PlaygroundMinAmount`. The backend currently authenticates only the organization-level bearer token — any terminal of that organization gets a playground ticket printed. We need admins to pick *which* terminals within an organization actually run the feature.

## Non-goals

- Per-terminal override of `min_amount` or the hardcoded `amount_per_child = 50000`. Out of scope; if needed later, add a dedicated settings table.
- Dedicated playground-settings admin page. A single boolean fits on the existing terminal edit form.

## Design

### 1. Database

Add a boolean column to `terminals`:

```ts
// backend/drizzle/schema.ts — inside pgTable("terminals", { ... })
playground_enabled: boolean("playground_enabled").default(false).notNull(),
```

Migration (generated by `drizzle-kit generate`):

```sql
ALTER TABLE "terminals"
  ADD COLUMN "playground_enabled" boolean DEFAULT false NOT NULL;

-- one-off backfill: preserve current behavior for existing terminals
UPDATE "terminals" SET "playground_enabled" = true;
```

Semantics:

- Default `false` for new terminals (explicit opt-in).
- Existing terminals are backfilled to `true` so currently-working installations are not broken by the migration.

### 2. Backend — `POST /api/playground_tickets/generate`

File: `backend/src/modules/playground_tickets/controller.ts`

Add a terminal lookup after the api-token block resolves `organization_id`, before `children_count` is computed:

```ts
const terminalRow = await drizzle
  .select({
    id: terminals.id,
    organization_id: terminals.organization_id,
    playground_enabled: terminals.playground_enabled,
  })
  .from(terminals)
  .where(eq(terminals.id, terminal_id))
  .execute();

if (terminalRow.length === 0 || terminalRow[0].organization_id !== organization_id) {
  set.status = 404;
  return { message: "Terminal not found" };
}

if (!terminalRow[0].playground_enabled) {
  set.status = 403;
  return { message: "Playground tickets disabled for this terminal" };
}
```

Response contract:

| Status | Meaning |
|--------|---------|
| 200    | Playground enabled; ticket created and returned. |
| 403    | Terminal exists but `playground_enabled = false`. Plugin treats this as "skip QR". |
| 404    | Terminal missing or belongs to a different organization than the token's org. |

Side benefit: the new 404 branch closes a pre-existing gap — the endpoint never verified that `terminal_id` belonged to the token's organization.

Pre-existing bug: `admin/app/[locale]/organization/terminals/columns.tsx:32` wires the terminal Edit button to `OrganizationsFormSheet` (the *organization* form) instead of the dedicated `TerminalsFormSheet`. Fix this as part of this change — otherwise the new toggle has no UI surface.

### 4. Plugin (`tablo_full/new_webserver`)

No code changes.

- `Helpers/PlaygroundTicketHelper.cs:49-53` already returns `null` when `HttpHelper.PostWithRetry` reports a non-success result (which covers the new 403 and 404 responses) and logs an error.
- `CustomWebserver.cs:426-430` already guards `if (ticket != null)` before calling `SetTicketForOrder`, so the cheque extender is never triggered for disabled terminals.
- Failures never block the sale — the whole block is wrapped in `try/catch` with "Never block sale due to playground ticket error" semantics.

## Build sequence

1. Schema + migration (`backend/drizzle/schema.ts` + `drizzle-kit generate`).
2. Backend terminals controller: extend POST and PUT body schemas.
3. Backend playground_tickets controller: add terminal lookup + enforcement.
4. Admin form: add `playground_enabled` switch.
5. Admin columns: swap to `TerminalsFormSheet`.
6. Run migration in staging; verify existing terminals are backfilled to `true`.
7. QA: toggle off a test terminal, confirm plugin receives 403 and does not print QR; toggle on, confirm QR prints.

## Risks / edge cases

- **Migration downtime:** `ALTER TABLE ... ADD COLUMN ... NOT NULL DEFAULT false` is a metadata-only change in modern PostgreSQL. The follow-up `UPDATE` touches every row; on a small `terminals` table (tens-hundreds of rows) this is negligible.
- **Cache:** `cacheController.cacheTerminals()` is already invoked by POST and PUT. After the migration, the first read hits the DB and repopulates the cache — no manual action needed.
- **Token → terminal mismatch attacks:** the new 404 check prevents a terminal with org A's `terminal_id` from being used when holding org B's token.
